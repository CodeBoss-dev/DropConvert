# Releasing a New Version

This is the playbook for shipping a new version of ConverterApp to users. Follow it top-to-bottom and you cannot mess it up.

## Context for Claude

ConverterApp is distributed via **GitHub Releases** at `https://github.com/CodeBoss-dev/ConverterApp/releases`. Each release contains:

- **`ConverterApp-<version>.zip`** — the runnable app. Users download this. ~200KB.
- **`LibreOffice-26.2.3-aarch64.tar.zst`** — the conversion engine. The app downloads this on first launch. ~137MB. Only re-uploaded when LibreOffice itself changes.

The download URL for the engine is hardcoded in `Sources/Utilities/LibreOfficeInstaller.swift`. The app .zip changes every release; the engine tarball almost never does.

## Versioning

Semantic versioning. Pick the right number:

- `1.0.0 → 1.0.1` — bug fix, no user-visible changes
- `1.0.0 → 1.1.0` — new feature, backwards-compatible
- `1.0.0 → 2.0.0` — breaking change (e.g., engine URL changed, file format incompatibility)

The current version lives in `Sources/Resources/Info.plist` as `CFBundleShortVersionString`.

## Prerequisites

Before running through the steps, verify these are true:

- [ ] You're on the `main` branch with no uncommitted changes other than the release work itself.
- [ ] `gh auth status` shows you're logged in to GitHub as `CodeBoss-dev`.
- [ ] **`GITHUB_TOKEN` env-var workaround:** the shell has a restricted env-var token that overrides the keychain. Every `gh` and `git push` command must be prefixed with `GITHUB_TOKEN=""` to force the keychain token. (Persistent fix: remove the `GITHUB_TOKEN` export from `~/.zshrc`. Not blocking.)
- [ ] The `LibreOffice/` folder at project root exists and is ~580MB stripped. Required only if you're rebuilding the engine tarball. If missing, run `bash Scripts/strip-libreoffice.sh` first.

## The 5-step release flow (app-only update)

This is the common case: you changed Swift code but **LibreOffice didn't change**. Engine tarball stays the same; only the .zip gets re-uploaded.

### Step 1 — Bump the version

```bash
cd /Users/waaridhborpujari/Documents/ConverterApp
NEW_VERSION="X.Y.Z"   # e.g. 1.2.0

plutil -replace CFBundleShortVersionString -string "$NEW_VERSION" Sources/Resources/Info.plist
plutil -replace CFBundleVersion -string "$NEW_VERSION" Sources/Resources/Info.plist

# Verify
plutil -p Sources/Resources/Info.plist | grep -i version
```

Expected: both `CFBundleShortVersionString` and `CFBundleVersion` show the new version.

### Step 2 — Commit and push the code

```bash
git status --short             # confirm only the intended files are modified
git add -A
git commit -m "Short description of changes

Longer body explaining what changed and why.

Bumps version to $NEW_VERSION."
GITHUB_TOKEN="" git push
```

### Step 3 — Build the app and package it

```bash
killall ConverterApp 2>/dev/null || true   # avoid "Resource busy" during rebuild
make bundle
rm -f "ConverterApp-${NEW_VERSION}.zip"
ditto -c -k --keepParent ConverterApp.app "ConverterApp-${NEW_VERSION}.zip"
ls -lh "ConverterApp-${NEW_VERSION}.zip"
```

**Critical:** use `ditto`, NOT `zip`. `ditto` preserves resource forks, code signatures, and extended attributes that regular `zip` strips — without them the unzipped app may refuse to launch with a vague Gatekeeper error.

Expected zip size: ~200-300 KB (just Swift code + Info.plist + ACKNOWLEDGMENTS).

### Step 4 — Publish the GitHub release

```bash
GITHUB_TOKEN="" gh release create "v${NEW_VERSION}" \
  "ConverterApp-${NEW_VERSION}.zip" \
  --title "ConverterApp ${NEW_VERSION}" \
  --notes "## What's new

- Feature one
- Feature two

## Download

**👉 [ConverterApp-${NEW_VERSION}.zip](https://github.com/CodeBoss-dev/ConverterApp/releases/download/v${NEW_VERSION}/ConverterApp-${NEW_VERSION}.zip)** — unzip and drag to your Applications folder.

The conversion engine from earlier releases is reused — no second download if you've used ConverterApp before.

## First launch

- Right-click the app → **Open** → confirm the security dialog. (macOS shows this for apps not from the App Store. You only need to do it once.)
- macOS 13 (Ventura) or later, Apple Silicon (M1/M2/M3/M4).

---

*The 'Source code' downloads below are auto-generated by GitHub — ignore them unless you want to build from source.*"
```

Expected output: a single URL like `https://github.com/CodeBoss-dev/ConverterApp/releases/tag/v1.2.0`. Open it in a browser to verify.

### Step 5 — Verify the live release

Manual sanity check before declaring it done:

1. Open the release URL printed in step 4.
2. Confirm `ConverterApp-${NEW_VERSION}.zip` is listed under "Assets" (not just the auto-generated source code).
3. Click the zip to verify it downloads.
4. (Optional) Unzip in `~/Downloads`, drag to a temp location, right-click → Open → run it to confirm the version-bumped app shows the right version in its menu (look for "Version X.Y.Z" in the dropdown).

## The full flow (engine also changed)

Only needed when LibreOffice itself updated (rare — security fix, format bug, etc.).

After step 1 (version bump), do this **before** step 2:

```bash
# Strip a fresh LibreOffice
bash Scripts/strip-libreoffice.sh

# Build new engine tarball (overrides old one)
rm -f LibreOffice-*.tar.zst
make engine-tarball

# Grab the new SHA-256 from the make output and update the installer
NEW_SHA=$(shasum -a 256 LibreOffice-*.tar.zst | awk '{print $1}')
echo "New engine SHA: $NEW_SHA"

# Manually edit Sources/Utilities/LibreOfficeInstaller.swift:
#   - update expectedSHA256 to the new value
#   - update downloadURL if the filename changed (it includes the LO version)
```

Then proceed with steps 2–4. In step 4, **also attach the engine tarball**:

```bash
GITHUB_TOKEN="" gh release create "v${NEW_VERSION}" \
  "ConverterApp-${NEW_VERSION}.zip" \
  "LibreOffice-26.2.3-aarch64.tar.zst" \
  --title "ConverterApp ${NEW_VERSION}" \
  --notes "..."
```

**Important:** if the engine version stays the same (e.g., LO 26.2.3) but you re-stripped it differently and the SHA changed, **every existing user's installed engine will fail SHA verification on first launch after they update**. The app currently does not re-download a mismatched engine — it errors out. Consider this carefully before pushing an engine change.

## Troubleshooting

### `HTTP 403: Resource not accessible by personal access token`

The shell's `GITHUB_TOKEN` env-var is overriding the keychain. Prefix the failing command with `GITHUB_TOKEN=""`.

### `failed to create release: name already exists`

The tag `vX.Y.Z` already exists (probably a botched earlier attempt). Two fixes:

```bash
# Option A — delete the existing release and tag, retry
GITHUB_TOKEN="" gh release delete "v${NEW_VERSION}" --yes
git push --delete origin "v${NEW_VERSION}"

# Option B — bump the patch version and try again (cleaner audit trail)
```

### `swift build` fails after version bump

`Info.plist` changes don't affect compilation. The build failure is unrelated — read the actual error.

### App won't launch after unzipping the new build

Almost always a `ditto` vs `zip` issue. Confirm step 3 used `ditto`. If yes, check the unzipped app's code signature:

```bash
codesign -dv ConverterApp.app
```

Should show `Signature=adhoc` and no errors.

### A user reports a conversion fails on a specific format

This is the case that *might* need an engine update. Triage first:

1. Reproduce locally. Drop the user's failing file on your dev build.
2. If the bug is in our Swift code (e.g., file detection, post-processing) — fix in Swift and ship a regular app-only update.
3. If the bug is genuinely inside LibreOffice — research whether a newer LO version fixes it. If yes, that's the "engine also changed" flow above. If no, document the limitation in the README.

## Why the engine is decoupled from the app

Worth understanding before you touch any of this. The app .zip is ~200 KB; the engine is ~137 MB. If they shipped together, every app update would re-download 137 MB even when nothing about the engine changed. By keeping them separate:

- **App updates are tiny.** Users on slow connections aren't punished for a UI tweak.
- **The engine is cached in `~/Library/Application Support/ConverterApp/LibreOffice/`** across app updates. The installer checks for it on launch and skips re-downloading if it's present and the SHA matches.
- **The engine URL is versioned** by *engine* version, not *app* version. `LibreOffice-26.2.3-aarch64.tar.zst` stays at that URL forever, even as app versions go 1.0 → 1.1 → 2.0. The app .zip and engine .tar.zst evolve on independent timelines.

This is the same architecture Xcode, Logic Pro, and similar large-asset apps use — the difference is they have Apple's update infrastructure and we have GitHub Releases. Same idea, smaller scale.

## What this doc deliberately does NOT cover

- **Sparkle auto-updates** — users currently find out about new versions by visiting the GitHub page. When that becomes annoying enough, add Sparkle. Not yet.
- **Notarization** — the app is ad-hoc signed and users see a Gatekeeper warning on first launch. Notarizing requires a $99/yr Apple Developer account. Defer until you have enough users that the friction matters.
- **Intel Mac support** — current build is arm64-only. Adding x86_64 means a universal build and a second engine tarball. Not yet.
- **Engine update notifications** — there's no in-app "new engine available" flow. If you ship a new engine, existing users won't pick it up until they reinstall the app. Fine while engine updates are rare; revisit if that changes.
